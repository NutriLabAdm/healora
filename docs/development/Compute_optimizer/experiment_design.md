# Эксперимент: снижение COMPUTE при накоплении ответов

## Гипотеза

По мере накопления пользовательских запросов и ответов оркестратор может снижать вычислительные затраты (COMPUTE) без потери качества (SCORE) за счёт:
- кэширования повторяющихся вопросов
- персонализации RAG-контекста
- сокращения глубины ансамбля для типовых сценариев

## Метрики

| Метрика | Цель | Порог успеха |
|---|---|---|
| SCORE | ≥ 85 | Среднее за эксперимент не ниже 85 |
| COMPUTE | ↓ от baseline | Снижение ≥ 40% к концу эксперимента |
| Hit rate кэша | ↑ | ≥ 60% после 500 запросов |
| Время ответа | ↓ | Снижение ≥ 50% p95 |

## Выборка

- База: 1000 вопросов по нутрициологии (`nutrition_questions.json`)
- 8 категорий, 4 уровня сложности
- Симуляция 10 000 запросов:
  - 70% — повторяющиеся паттерны (имитация активных пользователей)
  - 30% — уникальные вопросы (новые пользователи / редкие темы)

## Фазы эксперимента

### Фаза 1 — Холодный старт (запросы 1–200)

- Ансамбль из 5 моделей (GPT-5.5-vision, Claude 4.6, Gemini 2.0, DeepSeek, Qwen)
- Полный RAG-поиск по общей базе знаний
- **Ожидаемый COMPUTE**: 5x (baseline)
- **Ожидаемый SCORE**: 92–95

### Фаза 2 — Накопление кэша (запросы 201–1000)

- Включается кэширование повторяющихся вопросов (embeddings + exact match)
- Оркестратор начинает сокращать ансамбль для типовых запросов
- **Ожидаемый COMPUTE**: 3–4x
- **Ожидаемый SCORE**: 88–93

### Фаза 3 — Персонализация (запросы 1001–5000)

- Для пользователей с историей (5+ запросов) включается персональный RAG
- Digital Twin обогащает контекст
- Оркестратор использует RL-политику выбора агентов
- **Ожидаемый COMPUTE**: 1.5–2.5x
- **Ожидаемый SCORE**: 86–90

### Фаза 4 — Насыщение (запросы 5001–10000)

- Hit rate кэша > 60%
- RL-политика стабилизирована
- Ансамбль используется только для сложных / новых запросов
- **Ожидаемый COMPUTE**: 0.8–1.2x
- **Ожидаемый SCORE**: 85–88

## Архитектура

```
Запрос → Классификатор → Проверка кэша
  ├─ hit → возврат закэшированного ответа → COMPUTE: 0.1x
  └─ miss → Оценщик сложности
            ├─ low → 1 модель (GPT-5.5-vision) → COMPUTE: 1x
            ├─ medium → ансамбль 3 модели → COMPUTE: 3x
            └─ high → ансамбль 5 моделей → COMPUTE: 5x
                  → Результат → Кэширование
```

## Сбор данных

Для каждого запроса логируется:


## Анализ результатов

- Скользящее среднее COMPUTE (окно 100 запросов)
- Распределение SCORE по фазам
- Матрица hit rate по категориям
- Асимптота COMPUTE при насыщении кэша

## Критерии остановки

Эксперимент завершается досрочно, если:
- SCORE падает ниже 80 в течение 50+ запросов подряд
- Hit rate не растёт 3 фазы подряд
- COMPUTE не снижается 2 фазы подряд
