# Plan-Journal Architecture

> Архитектура формирования планов терапии с консистентностью и HITL
> Дата: 22.05.2026 (обновлено 23.05.2026 — cloud-архитектура)

## Cloud-архитектура Healora (Online/API-First)

Архитектура Healora в формате онлайн-сервиса реализуется как API-ориентированная распределённая система, обеспечивающая удалённый доступ к интеллектуальным медицинским функциям через веб-интерфейс, мобильные приложения и интеграционные шлюзы. В отличие от замкнутого on-premise решения, вычислительная инфраструктура и модели искусственного интеллекта размещаются в облачной среде, что обеспечивает масштабируемость, гибкость обновления и возможность подключения внешних медицинских сервисов.

Пользователь взаимодействует с системой через клиентское приложение, передавая данные о состоянии здоровья, образе жизни, лабораторных исследованиях, медицинской истории и иных параметрах. На основе полученной информации формируется **персональный цифровой двойник**, представляющий собой динамическую структуру медицинского профиля пользователя.

Данные через **API Gateway** передаются в интеллектуальное ядро системы (**AI Core**), включающее большие языковые модели (LLM), специализированных медицинских агентов и механизм Retrieval-Augmented Generation (RAG). Для обработки запросов используются внешние и внутренние **базы знаний** (Knowledge Layer), графы медицинских сущностей, векторные хранилища эмбеддингов, клинические протоколы и механизмы семантического поиска. Такой подход позволяет формировать персонализированные рекомендации, профилактические сценарии и варианты терапевтического сопровождения.

Дополнительно архитектура предусматривает отдельный контур интеграции с **внешними научными источниками** (External Sources) — медицинскими публикациями и клиническими руководствами. Новые данные проходят автоматический анализ, структурирование и **экспертную валидацию** перед включением в базу знаний.

Важным элементом системы является механизм **персонального хранения данных пользователя** (Personal Storage). Сформированный цифровой двойник, история изменений, рекомендации и утверждённые протоколы терапии могут сохраняться не только в защищённом облачном хранилище, но и экспортироваться на персональный носитель пользователя: зашифрованное мобильное устройство, локальное хранилище, персональное облако, медицинская карта, USB-носитель или цифровой контейнер пациента.

### Компоненты cloud-архитектуры

| Компонент | Кто выполняет | Модуль | Назначение / решаемая задача |
|---|---|---|---|
| **API Gateway** | API-шлюз и сервис авторизации | Интеграционный слой | Принимает запросы от веб-приложений, мобильных клиентов и внешних систем, выполняет аутентификацию, маршрутизацию и контроль доступа. Решает задачу безопасного взаимодействия компонентов системы. |
| **Цифровой двойник** | Модуль профиля пользователя | Data Layer | Формирует единую персональную модель пациента, объединяя демографию, анамнез, лабораторные показатели, данные образа жизни и витальные параметры. Решает задачу централизованного хранения медицинского контекста. |
| **LLM + RAG** | Языковые модели и интеллектуальные агенты | AI Core | Выполняет анализ персональных данных, поиск релевантной информации и генерацию рекомендаций. Решает задачу персонализации и интеллектуальной поддержки принятия решений. |
| **База знаний** | Система хранения и семантического поиска | Knowledge Layer | Хранит медицинские протоколы, граф знаний, эмбеддинги, лекарственные и пищевые взаимодействия. Решает задачу предоставления проверенных данных для рекомендаций. |
| **Экспертный контур** | Врач-эксперт и валидатор | Human-in-the-loop | Проверяет автоматически обработанную информацию и подтверждает клиническую корректность. Решает задачу повышения достоверности и снижения ошибок ИИ. |
| **Внешние источники** | Коннекторы и парсеры | Data Integration | Получает научные статьи, клинические рекомендации и нормативные документы. Решает задачу регулярного обновления медицинских знаний. |
| **Персональное хранилище** | Модуль экспорта и синхронизации | Personal Storage | Сохраняет цифровой двойник, историю наблюдений и протоколы терапии на персональном устройстве или защищённом контейнере пользователя. Решает задачу переносимости данных и контроля пользователя над информацией. |

## 1. Концепция

**Plan-Journal** — единый JSON-файл, который одновременно является планом терапии и журналом выполнения. Single source of truth для всех компонентов системы: чат, UI, аналитика, экспорт.

## 2. Схема plan.json

```json
{
  "plan_id": "uuid",
  "profile_id": "uuid",
  "protocol_id": "cognitive_health",
  "title": "Когнитивное здоровье — 14 дней",
  "status": "draft | active | paused | completed | cancelled",
  "created_at": "2026-05-21T12:00:00Z",
  "source": "engine | hitl",
  "approved_by": null,
  "schedule": [
    {
      "id": "int_001",
      "day": 1,
      "date": "2026-05-22",
      "code": "MN_MDT",
      "type": "meditation",
      "scheduled_time": "08:00",
      "params": { "duration": 10, "audio": "calm_ocean" },
      "status": "pending | done | skipped | modified | rescheduled | cancelled",
      "result": null,
      "completed_at": null,
      "comment": null
    }
  ],
  "computed": {
    "progress_pct": 45,
    "done_count": 4,
    "total_count": 14,
    "streak_days": 3,
    "missed_streak": 0,
    "next_intervention": "2026-05-22T08:00:00Z"
  },
  "changelog": [
    { "at": "2026-05-21T12:00:00Z", "action": "created", "by": "engine" }
  ]
}
```

## 3. Принципы

### 3.1. Single source of truth
- Все компоненты читают только этот файл
- Никаких дублирующих состояний в localStorage, Redis и т.д.
- При изменении статуса интервенции — файл перезаписывается целиком

### 3.2. Консистентность
- Пишет только API сервер (1 процесс, без конкурентного доступа)
- Lock-файл не нужен (однопоточная запись)
- При рестарте сервера — план перечитывается с диска

### 3.3. Портативность
- Файл можно скачать, отправить в Telegram, перенести на флешку
- Любой viewer (или текстовый редактор) показывает актуальное состояние
- Plan-Journal сам содержит computed-поля (progress, streak) — метрики не теряются при переносе

### 3.4. Версионность
- Пока не нужна. В крайнем случае — git (файлы в репозитории)

## 4. Поток генерации

```
User Goal → [Goal Parser] → [Protocol Matcher] → [Schedule Generator]
                                                       │
                                                       ▼
                                              plan.json (draft)
                                                       │
                                              ┌────────┴────────┐
                                              │ user wants HITL │
                                              └────────┬────────┘
                                                  yes     no
                                                   │      │
                                                   ▼      ▼
                                              HITL апрув  active
                                              (оплачено)  (бесплатно)
```

### 4.1. Goal Parser
- Пользователь видит предопределённые бейджи (улучшить сон, снизить вес, когнитивное здоровье, снизить стресс и т.д.)
- Или свободный текстовый ввод → `POST /api/goal/parse` → LLM задаёт уточняющие вопросы, парсит ответы
- LLM возвращает предложение: бейджи с протоколами, практиками, интервенциями
- Пользователь выбирает подходящий → запускается Schedule Generator

### 4.2. Protocol Matcher
- По цели → поиск подходящего протокола (RAG по каталогу протоколов)
- Учёт профиля пользователя (лабы, противопоказания)
- Возвращает protocol_id

### 4.3. Schedule Generator
- По protocol_id → загрузка шаблона расписания из протокола
- Адаптация под пользователя (интенсивность, длительность)
- Генерация plan.json со статусом `draft`

### 4.4. HITL (Human-in-the-Loop)
- **Бесплатно:** план сразу active, пользователь может корректировать вручную
- **HITL (оплачено):** план уходит на апрув специалисту → тот утверждает или правит → plan.json становится active
- Статус `approved_by` фиксирует, кто апрувил

## 5. Поток выполнения

```
Ежедневно:
  1. UI читает plan.json → показывает сегодняшние интервенции
  2. User выполняет → тыкает "done"
  3. API: PATCH /plan/:id/intervention/:int_id { status, result, comment }
  4. API: обновляет plan.json
  5. API: пересчитывает computed (progress, streak)

При пропуске:
  1. API записывает skipped
  2. Система предлагает сдвинуть расписание
  3. User решает: сдвинуть или пропустить

Напоминания:
  1. Генерируются из plan.json (scheduled_time)
  2. Не генерируются на ходу — просто просмотр файла
```

## 6. Endpoints API

| Метод | Путь | Описание |
|---|---|---|
| `POST` | `/api/plan/generate` | Создать plan.json (draft) |
| `GET` | `/api/plan/:plan_id` | Получить plan.json |
| `PATCH` | `/api/plan/:plan_id/intervention/:int_id` | Обновить статус интервенции |
| `POST` | `/api/plan/:plan_id/approve` | HITL апрув (draft → active) |
| `POST` | `/api/plan/:plan_id/reschedule` | Сдвиг интервенции |
| `GET` | `/api/plan/:plan_id/export` | Скачать plan.json |
| `GET` | `/api/plans?profile_id=...` | Список планов пользователя |
| `POST` | `/api/goal/parse` | LLM-парсинг цели → предложение бейджей, протоколов |

## 7. Связь с RAG

### Стратегия: Hybrid

**Как работает:**

1. **Классификация интента** — для каждого запроса определяем тип:
   - `context` — вопросы по плану, статусу, прогрессу → inject plan.json в system prompt
   - `general` — общие Q&A → RAG без плана
   - `status` — отметка выполнения → API запись без LLM

2. **Plan injection** — только для 10–20% запросов (вопросы про план)

**Затраты токенов:**
- plan.json в промпте: ~300–800 токенов, только на context-запросах
- 50 запросов/день → 5k–15k токенов/день

**Плюсы:** баланс затрат и качества, план в контексте только когда нужен

## 8. Хранение

- Путь: `api/user_data/plans/{user_id}_{plan_id}_{plan_type}.json`
  - `user_id` — UUID пользователя (группировка по папкам не нужна, префикс достаточно)
  - `plan_id` — уникальный ID плана
  - `plan_type` — тип плана (например, `cognitive`, `sleep`, `weight`)
- Путь настраивается через `PLANS_DIR` в .env (можно указать путь на флешку)
- **In-memory cache**: при старте сервера все plan.json читаются в Map. Каждый write пишет и в Map, и на диск (sync). Чтение — только из памяти
- **Синхронизация**: write всегда синхронный на диск (fs.writeFileSync), чтобы при краше данные не терялись

## 9. Каталог интервенций

`interventions_catalog.json` расширяется полями `allowed_statuses` и `params_schema` для каждого кода:
```json
{
  "code": "PH_HIIT",
  "allowed_statuses": ["pending", "done", "skipped", "modified", "rescheduled", "cancelled"],
  "params_schema": {
    "duration": { "type": "number", "min": 5, "max": 120 },
    "intensity": { "type": "string", "enum": ["low", "medium", "high"] }
  }
}
```

## 10. Метрики

Computed-поля в самом plan.json:
- `progress_pct` — % выполненных интервенций
- `done_count / total_count` — счётчики
- `streak_days` —连续 дней выполнения
- `missed_streak` —连续 пропусков
- `next_intervention` — ближайшая запланированная

Это позволяет скопировать файл и продолжить в другом месте или учётной записи.
