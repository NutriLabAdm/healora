# 3.3. Метод формирования персонализированных наборов интервенций на основе регрессионных моделей

> **Резюме для заявки в Сколково**
>
> Разработан математический аппарат и алгоритм генерации персонализированных наборов интервенций для цифрового двойника пациента с ожирением. Метод преобразует протокол терапии (средиземноморская диета, кетодиета, GLP-1RA и др.) в структурированный набор `S = {(INT_i, w_i, f_i, r_i)}` с тремя параметрами на каждую интервенцию: важность (w_i, 5-факторная взвешенная сумма: доказательность, согласование с целями, совместимость с DT, регрессионный.score, снятие барьеров), интенсивность (f_i, адаптация по TTM-стадии) и регулярность (r_i, подстройка под расписание). Предусмотрены 3 технологические реализации с каскадной архитектурой: **RAG-пайплайн** (основной канал, интерпретируемые предсказания с 95% доверительными интервалами и тремя сценариями прогноза), **регрессионная модель ElasticNet** (fallback, ансамбль NHANES+BRFSS+Synthea+пользовательские логи), **deep prompting** (экстренный zero-shot через 4-слойный промптинг). Правки пользователя сохраняются как train-data для дообучения. Дорожная карта: 4 фазы от Q4 2026 до Q3 2027.

## 1. Постановка задачи

Предсказать целевой признак **Z** (эффективность интервенции, adherence, изменение веса) с помощью регрессии на основе набора генетических мутаций, данных о метилировании ДНК и фенотипических характеристик пользователя.

## 2. Математическая модель

```
Z = β₀ + β₁·X₁ + β₂·X₂ + … + β_k·X_k +
         + γ₁·M₁ + γ₂·M₂ + … + γ_m·M_m +
         + δ₁·A₁ + δ₂·A₂ + … + δ_a·A_a + ε
```

где:
- **Z** — предсказанный признак (adherence, response, weight delta)
- **X_i** — генетические мутации (SNP генов FTO, MC4R, PPARG, TCF7L2 и др.)
- **M_j** — уровень метилирования CpG-участков (0–100%), динамический эпигенетический маркер
- **A_p** — фенотип (вес, рост, возраст, пол, уровень активности)
- **β, γ, δ** — коэффициенты регрессии (вклад каждого предиктора)
- **ε** — случайный шум

## 3. Учёт метилирования ДНК

Метилирование ДНК — обратимая модификация, меняющаяся под действием питания, образа жизни и стресса. Гиперметилирование промоторов → подавление экспрессии генов. Ключевые гены-кандидаты: FTO (аппетит), MC4R (насыщение), PPARG (липиды), LEP/LEPR (лептин), IGF2 (метаболическое программирование).

## 4. Типы регрессионных моделей

| Модель | Назначение | Регуляризация |
|--------|-----------|---------------|
| **Linear Regression** | Базовое предсказание | — |
| **Ridge (L2)** | Мультиколлинеарность SNP | α настраивается GridSearchCV |
| **ElasticNet (L1+L2)** | Отбор значимых признаков (SNP + CpG) | l1_ratio, α |
| **Logistic Regression** | Бинарный отклик (есть/нет) | L1, solver='saga' |

## 5. Метод преобразования протоколов терапии в персонализированный набор интервенций

### 5.1. Постановка задачи

На входе: протокол терапии (например, "средиземноморская диета для снижения веса"), цели пользователя, параметры Digital Twin.
На выходе: структурированный набор интервенций `S = {(INT_i, w_i, f_i, r_i)}`, где:

| Параметр | Обозначение | Описание |
|----------|------------|----------|
| Интервенция | INT_i | Конкретное действие из каталога (03.2) |
| Важность применения | w_i ∈ [0, 1] | Коэффициент значимости данной интервенции для достижения целей |
| Интенсивность | f_i ∈ [0.25, 2.0] | Множитель интенсивности (0.25 = минимальная, 1.0 = стандартная, 2.0 = максимальная) |
| Регулярность | r_i | Частота применения: daily / weekly / monthly / on-demand |

### 5.2. Формальная модель преобразования

#### 5.2.1. Базовое правило отбора

Для каждого протокола терапии P и набора целей G = {g_1, ..., g_n} определяется начальный пул интервенций:

```
Candidates(P, G) = { INT_i ∈ Catalog | match(INT_i.category, P.domains) > θ_match 
                                         AND relevance(INT_i, G) > θ_rel }
```

где:
- `match(INT_i.category, P.domains)` — пересечение категорий интервенции с доменами протокола (0–1)
- `relevance(INT_i, G)` — релевантность интервенции набору целей (0–1), вычисляется через RAG-близость эмбеддингов
- `θ_match, θ_rel` — пороговые значения (по умолчанию 0.3 и 0.5)

#### 5.2.2. Расчёт важности применения w_i

Важность применения w_i для интервенции INT_i рассчитывается как взвешенная сумма пяти факторов:

```
w_i = α₁ · evidence(INT_i) 
    + α₂ · goal_alignment(INT_i, G) 
    + α₃ · twin_compatibility(INT_i, DT) 
    + α₄ · regression_score(INT_i, Z) 
    + α₅ · barrier_mitigation(INT_i, DT.barriers)
```

| Фактор | Вес α | Описание |
|--------|-------|----------|
| `evidence(INT_i)` | 0.25 | Доказательность: A=1.0, B=0.7, C=0.4, D=0.1 |
| `goal_alignment(INT_i, G)` | 0.25 | Косинусная близость эмбеддинга INT_i к вектору целей |
| `twin_compatibility(INT_i, DT)` | 0.20 | Совместимость с профилем (возраст, пол, ИМТ, активность) |
| `regression_score(INT_i, Z)` | 0.20 | Предсказание регрессионной модели Z для данной интервенции |
| `barrier_mitigation(INT_i, DT.barriers)` | 0.10 | Степень снятия барьеров пользователя (0–1) |

Итоговый коэффициент нормализуется: `w_i ∈ [0, 1]`.

#### 5.2.3. Расчёт интенсивности f_i

Интенсивность зависит от стадии готовности пользователя (TTM — Transtheoretical Model):

```
f_i = base_intensity(INT_i) × readiness_modifier(DT.stage)
```

| Стадия TTM | readiness_modifier | Описание |
|------------|-------------------|----------|
| Precontemplation | 0.25 | Минимальная интенсивность, ознакомление |
| Contemplation | 0.50 | Низкая интенсивность, пробные шаги |
| Preparation | 0.75 | Умеренная интенсивность |
| Action | 1.00 | Стандартная интенсивность |
| Maintenance | 0.85 | Поддерживающая интенсивность (снижение) |

#### 5.2.4. Расчёт регулярности r_i

Регулярность определяется типом интервенции и адаптируется под расписание пользователя через DT.preferences:

```
r_i = default_regularity(INT_i) × schedule_adjust(DT.preferences.time_slots)
```

| Тип интервенции | default_regularity | Пример |
|-----------------|-------------------|--------|
| Поведенческая привычка | daily | "10 мин медитации" |
| Тренировка | weekly (3–5 раз) | "HIIT 3 раза в неделю" |
| Медицинский скрининг | monthly / yearly | "Чекап раз в год" |
| Образовательная | on-demand | "Чтение статьи" |

### 5.3. Алгоритм генерации набора

```
Вход:  протокол P, цели G, цифровой двойник DT
Выход: набор интервенций S

1. Candidates = filter_catalog(P, G)          // отбор из каталога
2. for each INT_i in Candidates:
3.     w_i = calculate_importance(INT_i, G, DT) // важность
4.     f_i = calculate_intensity(INT_i, DT)      // интенсивность
5.     r_i = calculate_regularity(INT_i, DT)     // регулярность
6. S = sort_by_importance(Candidates)            // сортировка по w_i
7. S = select_top_k(S, K=max_interventions)      // топ-K интервенций
8. S = apply_constraints(S, DT.constraints)       // учёт ограничений
9. return S
```

### 5.5. Учёт динамики (адаптация во времени)

Набор интервенций S не статичен — он пересчитывается по расписанию:

| Триггер | Период | Действие |
|---------|--------|----------|
| Плановый | 7 дней | Пересчёт w_i, f_i, r_i на основе новых данных трекинга |
| Достижение цели | событийно | Снижение f_i для достигнутых целей, добавление новых |
| Смена стадии TTM | событийно | Повышение f_i при переходе к Action |
| Низкий adherence | 3 дня | Снижение f_i, упрощение протокола |
| Обновление DT | событийно | Пересчёт twin_compatibility при изменении DT |

### 5.6. Ограничения и правила безопасности

```
apply_constraints(S, DT.constraints):
  1. Исключить INT_i с contraindications, пересекающимися с DT.conditions
  2. Ограничить суммарную нагрузку: Σ(w_i · f_i) < MAX_DAILY_BURDEN
  3. Баланс категорий: не более 40% интервенций из одной категории
  4. Минимальный набор: не менее 3 интервенций, не более 12
```

## 6. Технологические реализации

Формальная модель (раздел 5) может быть реализована тремя технологическими подходами — от простого к сложному. Каждый подход покрывает полный цикл: отбор кандидатов → расчёт w_i/f_i/r_i → сортировка → top-K → ограничения.

---

### 6.1. Уровень 1: Deep prompting + сократический промптинг (L1)

**Идея:** LLM (GPT-4o, Claude, YandexGPT) генерирует набор интервенций через 4-слойный промптинг, валидированный в NIOKR: Constitution → Context → Constraints → Conditioning. LLM выступает и как «регрессионная модель» (неявно вычисляя w_i через chain-of-thought), и как «RAG-ретривер» (используя knowledge cutoff + few-shot).

**Архитектура промпта:**

```
┌────────────────────────────────────────────┐
│  L1: Constitution                          │
│  «Ты — AI-специалист по персонализиро-    │
│   ванным интервенциям в области питания    │
│   и метаболизма. Ответ — JSON.»            │
├────────────────────────────────────────────┤
│  L2: Context                               │
│  Протокол P: {название, domains, duration} │
│  Цели G: {weight_loss: 5kg, ...}           │
│  DT: {возраст, ИМТ, TTM_stage, барьеры,   │
│       генотип FTO/MC4R/PPARG, метилир-ние}│
├────────────────────────────────────────────┤
│  L3: Constraints                           │
│  - Формат: [{code, w, f, r, reasoning}]   │
│  - 3 ≤ N ≤ 12, не более 40% из 1 катег.   │
│  - f_i = base × readiness(TTM)            │
│  - w_i = Σα·factor с пояснением вклада   │
├────────────────────────────────────────────┤
│  L4: Conditioning                          │
│  «Проверь: нет ли противопоказаний по     │
│   DT.conditions. Размышляй step-by-step.»  │
└────────────────────────────────────────────┘
```

**Сократический диалог (после генерации):**

```
Пользователь: «Сделай проще, у меня нет времени на HIIT»
LLM: → снижает f_i(HIIT) до 0.5, предлагает замену на ходьбу (LISS)
    → объясняет: «HIIT заменён на LISS (w=0.65). 
       Сэкономлено 25 мин/день. Прогноз Z: −1.8 кг за 4 нед»
```

**Оценка:**

| Критерий | Показатель |
|----------|-----------|
| Время внедрения | 2–4 недели |
| Точность w_i | Зависит от LLM; без fine-tuning ~70–80% от эталона |
| Стоимость инференса | $0.01–0.05 за запрос (GPT-4o mini); $0.10–0.30 (GPT-4o) |
| Данные для обучения | Не требуются (zero-shot) |
| Интерпретируемость | Средняя (chain-of-thought, но без численных весов) |
| Основной риск | Галлюцинации, нестабильность ответов, дрейф модели |

---

### 6.2. Уровень 2: Регрессионная модель (Ridge/ElasticNet) + ансамбль датасетов (L2)

**Идея:** Тренируем регрессионную модель для предсказания Z (adherence / weight delta) на ансамбле публичных медицинских когорт + синтетических данных. Коэффициенты β модели используются как `regression_score(INT_i, Z)` в формуле w_i. Ансамбль датасетов:

| Источник | Размер | Признаки | Целевая |
|----------|--------|----------|---------|
| **NHANES** (CDC) | 10 000+ | Фенотип, биомаркеры, диета | ИМТ, Δвес |
| **BRFSS** (CDC) | 400 000+ | Фенотип, поведение, хронич. заболевания | adherence к диете |
| **Synthea** (синтетика) | 1 000 000+ | Полный профиль + генетика (имитация SNP) | adherence + Δвес |
| **Пользовательские логи** | N (растёт) | Трекинг, f_i, фактические INT | real adherence (post-hoc) |

**Архитектура:**

```
┌──────────────┐   ┌──────────────┐   ┌──────────────┐
│  NHANES      │   │  BRFSS       │   │  Synthea     │
│  (претрен.)  │   │  (претрен.)  │   │  (претрен.)  │
└──────┬───────┘   └──────┬───────┘   └──────┬───────┘
       │                  │                  │
       └──────────────────┼──────────────────┘
                          ▼
                 ┌──────────────────┐
                 │  Base model      │
                 │  ElasticNet      │
                 │  (α, l1_ratio   │
                 │  GridSearchCV)   │
                 └──────┬───────────┘
                        │
                 ┌──────▼───────────┐
                 │  Fine-tune       │
                 │  на пользоват.   │
                 │  данных (logs)   │
                 └──────┬───────────┘
                        │
                 ┌──────▼───────────┐
                 │  inference:      │
                 │  Z = β₀ + ΣβX   │
                 │  + ΣγM + ΣδA    │
                 └──────────────────┘
```

**Отбор признаков (ElasticNet):** L1-регуляризация обнуляет незначимые признаки. Для 3 сценариев (A/B/C):

| Сценарий | Значимые признаки | Топ-1 |
|----------|------------------|-------|
| A (FTO AT, мет. 72%) | FTO, TCF7L2, ИМТ, активность | FTO.AT β=0.31 |
| B (MC4R, мет. 15%) | MC4R, LEP, возраст, пол | MC4R.C β=0.42 |
| C (PPARG, мет. 85%; LEPR, мет. 78%) | PPARG, LEPR, KCNQ1, ИМТ, возраст | PPARG.G β=0.38 |

**Оценка:**

| Критерий | Показатель |
|----------|-----------|
| Время внедрения | 4–8 недель (сбор данных + обучение) |
| Точность w_i | R² ~0.6–0.8 на тестовой выборке |
| Стоимость инференса | ~$0.0001 за запрос (CPU inference) |
| Данные для обучения | 1000+ профилей для fine-tuning |
| Интерпретируемость | Высокая (коэффициенты β напрямую читаемы) |
| Основной риск | Холодный старт; обобщаемость на новые SNP/фенотипы |

### 6.3. Уровень 3: RAG-пайплайн с интервалами неопределённости (L3)

**Идея:** Ретривер извлекает релевантные фрагменты из `protocol_obecity` (105 источников, S001–S105) + PubMed по каждому INT_i. LLM/модель вычисляет Z и доверительный интервал на основе размера эффекта и дисперсии в релевантных исследованиях.

**Архитектура:**

```
                ┌────────────────────────────┐
                │  Пользовательский запрос    │
                │  INT_i + DT params          │
                └────────────┬───────────────┘
                             ▼
                ┌────────────────────────────┐
                │  Query encoder             │
                │  (e5-large-v2 / bge-ru)    │
                └────────────┬───────────────┘
                             ▼
           ┌──────────────────────────────────┐
           │  Vector DB (Qdrant / Chroma)     │
           │  ┌────────────────────────────┐  │
           │  │ protocol_obecity (S001–105)│  │
           │  │ PubMed abstracts (быстрое  │  │
           │  │  индексирование)           │  │
           │  │ Онboarding данные          │  │
           │  └────────────────────────────┘  │
           └────────────────┬─────────────────┘
                            ▼
                ┌────────────────────────────┐
                │  Retrieve top-K chunks     │
                │  per INT_i                 │
                └────────────┬───────────────┘
                             ▼
                ┌────────────────────────────┐
                │  Compute Z + CI            │
                │                            │
                │  Z_i = Σ(effect_size_j ·   │
                │        similarity_j) /     │
                │        Σsimilarity_j       │
                │                            │
                │  CI_i = weighted_std(Z_i)   │
                │        × 1.96              │
                └────────────┬───────────────┘
                             ▼
                ┌────────────────────────────┐
                │  Формирование ответа       │
                │  INT_i: w_i, f_i, r_i     │
                │  Z_i = μ ± σ              │
                │  scenario: [оптим, реалис,│
                │   пессим]                 │
                └────────────────────────────┘
```

**Расчёт Z и интервала:**

```
Для каждого INT_i:
  Chunks = retrieve(INT_i, DT.factors)    // top-5~10 чанков
  Z_i = Σ(effect_size_j · sim_j) / Σsim_j // взвешенное среднее
  σ²_i = Σ(sim_j · (effect_j − Z_i)²) / Σsim_j  // взвешенная дисперсия
  CI_i = 1.96 · σ_i                        // 95% доверительный интервал

  Сценарии (для пользователя):
    optimistic = Z_i + σ_i
    realistic  = Z_i
    pessimistic = Z_i − σ_i
```

**Оценка:**

| Критерий | Показатель |
|----------|-----------|
| Время внедрения | 8–16 недель (индексация + пайплайн) |
| Точность w_i | Зависит от покрытия векторной БД; при полном покрытии — эталон |
| Стоимость инференса | $0.005–0.03 за запрос (embedding + LLM) |
| Данные для обучения | Не требуются; нужна качественная векторная БД |
| Интерпретируемость | Максимальная (каждый вклад ссылается на источник) |
| Основной риск | Латенси (2–5 сек на запрос); качество ретрива; устаревание БД |

### 6.4. Каскадная архитектура (L1 → L2 → L3)

Выбран гибридный подход: **каскад от простого к сложному** с эскалацией при недостатке данных.

```
Вход: запрос пользователя
  │
  ▼
┌────────────────────────────────────────────────┐
│  RAG (L3) — основной канал                    │
│  Если в БД достаточно evidence по INT_i:      │
│    → Z_i, CI_i, 3 сценария                    │
│    → w_i по 5 факторам с evidence из БД       │
│  ИНАЧЕ (холодный старт / новый INT):         │
│    → спускаемся на L2                          │
└────────────────────┬───────────────────────────┘
                     │
                     ▼
┌────────────────────────────────────────────────┐
│  Регрессия (L2) — fallback                    │
│  Если фенотип покрывается обученной моделью:  │
│    → Z_i из β-коэффициентов                    │
│    → CI_i из residual std                      │
│  ИНАЧЕ (новый SNP / крайний фенотип):        │
│    → спускаемся на L1                          │
└────────────────────┬───────────────────────────┘
                     │
                     ▼
┌────────────────────────────────────────────────┐
│  Deep prompting (L1) — экстренный fallback    │
│  LLM генерирует набор zero-shot               │
│  → без CI, с CoT-обоснованием                 │
│  → результат сохраняется в лог для дообучения │
│    L2-модели                                   │
└────────────────────────────────────────────────┘
```

**Взаимодействие с пользователем (dialog, ответ на вопросы):**

Пользователь вносит правки → правки сохраняются как train-data:
- Если правка меняет w_i → запись в лог `{INT_i, DT, user_w_i, llm_w_i}`
- Накопив 100+ правок → дообучение L2-модели
- Накопив 1000+ → переиндексация L3 с учётом пользовательских эффектов
