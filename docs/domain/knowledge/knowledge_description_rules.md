# Knowledge Description Rules

> Правила структурирования документов в каталоге `docs/domain/knowledge/`.

## 1. Общие принципы

- Каждый раздел знаний — это **папка** в `docs/domain/knowledge/` с говорящим английским названием (lowercase, underscore).
- В каждой папке обязателен `_index.md` — оглавление, описание и карта раздела.
- Документы пишутся на **русском языке** с сохранением английских терминов (protocol_id, evidence_level, tags).
- Машинно-читаемые данные дублируются в `data.json` (для приложения) и markdown (для человека).
- Каждый документ имеет **YAML frontmatter** с метаданными.

## 2. Структура папки раздела

```
section_name/
├── _index.md           # Описание, таблица содержимого, связи
├── data.json           # (опционально) Машинно-читаемый каталог
├── 01_topic.md         # Нумерованные темы
├── 02_topic.md
└── ...                 # Подпапки для подразделов
```

### 2.1 `_index.md`

```yaml
---
title: Название раздела
description: Краткое описание (1-2 предложения)
sources:
  - Автор — Название источника
  - Другой источник
tags: [раздел, тематика]
created: 2026-05-27
updated: 2026-05-27
---
```

- Содержит таблицу содержимого: `# | Файл | Описание`
- Содержит раздел «Связанные документы» с относительными ссылками на смежные разделы
- Может содержать концептуальные схемы (ASCII-диаграммы)

### 2.2 Тематический документ

```yaml
---
title: Название темы
description: Краткое описание (1-2 предложения)
source: Автор — Источник (год)
tags: [тег1, тег2]
status: draft | review | done
evidence_level: high | medium | low
created: 2026-05-27
updated: 2026-05-27
---
```

## 3. Naming conventions

| Сущность | Правило | Пример |
|----------|---------|--------|
| Папка раздела | `lowercase_with_underscore` | `environment_design/`, `diet_preferences/` |
| `_index.md` | Символ `_` в начале | `_index.md` |
| Тематический документ | `[номер_]название.md` | `01_atomic_habits.md`, `glycemic_control.md` |
| Подпапка | `lowercase_with_underscore` | `protocol_obecity/behavioral/` |
| data.json | Всегда `data.json` | `data.json` |
| Изображения | `нижний_регистр_латиница.расширение` | `04._яишница_с_овощами.jpg` |

### 3.1 Допустимые расширения

- `.md` — основной формат
- `.json` — машинно-читаемые данные
- `.jpg`, `.png` — иллюстрации
- `.html` — **только** для самостоятельных страниц (напр. `supplements/index.html`)

## 4. Frontmatter — обязательные поля

```yaml
---
title: string          # Заголовок документа (обязательно)
description: string    # Краткое описание, 1-2 предложения (обязательно)
---
```

### 4.1 Рекомендуемые поля

```yaml
source: string | string[]    # Автор — Название источника (год)
tags: string[]               # Массив тегов
status: draft | review | done
evidence_level: high | medium | low
created: YYYY-MM-DD
updated: YYYY-MM-DD
```

### 4.2 Поля для протоколов (`protocol_obecity/`, `protocols/`)

```yaml
id: string                # Уникальный ID протокола
name: string              # Название на английском
name_ru: string           # Название на русском
category: string          # behavioral | dietary | pharmaceutical | etc.
tags: string[]
status: draft | review | done
evidence_level: high | medium | low
created: YYYY-MM-DD
updated: YYYY-MM-DD
sources: string[]         # Массив ID источников из _relevance_table.md
```

## 5. Структура документа

### 5.1 Базовая структура тематического документа

```markdown
# Заголовок H1

**Ключевое поле:** Значение
**Другое поле:** Значение

## Введение / Что это такое
1-2 абзаца описания.

## Основное содержание
- Таблицы для сравнения
- Маркированные списки
- ASCII-диаграммы для концептуальных моделей

---

## Разделы с ---
Длинные документы разделяются горизонтальной чертой между крупными секциями.

---

## Источники и кросс-ссылки
- `folder/file.md` — описание связи
- `folder/file.md#section` — ссылка на конкретную секцию
```

### 5.2 Маркдаун-правила

| Элемент | Формат | Пример |
|---------|--------|--------|
| Заголовки | `# H1`, `## H2`, `### H3` | `# Витамин A` |
| Таблицы | GitHub Flavored Markdown | `\| Колонка 1 \| Колонка 2 \|` |
| Жирный текст | `**текст**` | `**Важно:**` |
| Курсив | `*текст*` | `*может варьироваться*` |
| Код | `` `код` `` | `` `protocol_id` `` |
| Блок кода | ` ``` ` | ` ```json\n{"key":"value"}\n``` ` |
| Цитаты | `> текст` | `> **Вывод:** 95% поведения...` |
| Горизонтальная черта | `---` | Между разделами |
| Ссылки | `[текст](путь)` | `[01_atomic_habits.md](01_atomic_habits.md)` |
| Изображения | `![alt](путь)` | `![Схема](schema.png)` |
| Сноски | `[^1]` | редко, только при необходимости |

## 6. Кросс-ссылки между разделами

- Использовать **относительные пути** от `docs/domain/knowledge/`.
- Формат: `folder/file.md` или `folder/file.md#section-name`.
- Пример: `psyhotypes/04_cjm_therapy_plans.md` — адаптация дизайна среды под психотип.
- В конце каждого `_index.md` добавлять блок **«Связанные документы»** со списком кросс-ссылок.

```
## Связанные документы
- `intervention/Каталог интервенций.md` — полный каталог вмешательств
- `psyhotypes/06_02_behavior.md` — таблица приёмов по 7 факторам поведения
```

## 7. Машинно-читаемые данные

- Каждый раздел может содержать `data.json` для использования в приложении.
- JSON должен содержать **те же данные**, что и markdown-документы, но в структурированном виде.
- Поля в JSON — на английском, значения — на русском (с дублированием `name_en` где нужно).
- Схема JSON документируется в `__SCHEMA_EXAMPLE.json` если сложная.

### Пример схемы data.json

```json
[
  {
    "id": "vit_a",
    "name_ru": "Витамин A",
    "name_en": "Vitamin A",
    "category": "vitamin",
    "type": "fat_soluble",
    "rda_rf": { "male": 900, "female": 700 },
    "functions": ["Зрение", "Иммунитет"],
    "top_foods": [
      { "name_ru": "Печень говяжья", "amount": 6500, "unit": "mcg" }
    ]
  }
]
```

## 8. Статусы документов

| Статус | Значение |
|--------|----------|
| `draft` | Черновик, может быть неполным |
| `review` | На ревью, ждёт проверки |
| `done` | Готов, прошёл ревью |

## 9. Типы разделов

| Тип | Описание | Пример |
|-----|----------|--------|
| **knowledge** | Справочные знания (витамины, минералы, БАДы) | `vitamins/`, `supplements/` |
| **protocol** | Протоколы терапии, клиентский путь | `protocols/`, `protocol_obecity/` |
| **catalog** | Каталоги интервенций/инструментов | `intervention/` |
| **guide** | Методологические руководства | `environment_design/`, `psyhotypes/` |
| **reference** | Справочные данные (таблицы, референсы) | `minerals/`, `food/` |
| **template** | Шаблоны планов лечения | `treatment_plans/` |
| **practice** | Практики (например, медицинские традиции) | `med_traditional_practices/` |
| **cases** | Наглядные примеры | `cases/` |

## 10. Проверка качества

Перед добавлением документа проверьте:

- [ ] Есть YAML frontmatter с `title` и `description`
- [ ] Файл лежит в правильной папке
- [ ] Название файла по соглашению (lowercase, underscore)
- [ ] `_index.md` обновлён (если это новый документ в существующем разделе)
- [ ] Ссылки на другие документы — относительные и рабочие
- [ ] JSON (если есть) синхронизирован с markdown
- [ ] Нет дублирования с другими разделами (проверьте кросс-ссылки)
- [ ] Дата `updated` актуальна

## 11. Исключения

- `supplements/index.html` — исключение: самостоятельное SPA-приложение, не следует правилам markdown.
- `food/*.jpg` — изображения, не документы.
- `product/PRODUCT_DESCRIPTION.md` — документ продукта, а не знаний.
- README.md — техническая документация, не знания.
