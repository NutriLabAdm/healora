## Description

Протокол описывает **ядро принятия решений Healora** — алгоритм, который на основе заполненных параметров цифрового двойника (DT):

1. Оценивает вероятность успеха для каждого доступного протокола/интервенции
2. Ранжирует их и показывает пользователю топ-3 с обоснованием
3. Итеративно запрашивает недостающие данные в порядке убывания коэффициента «снижение неопределённости / усилие»

Протокол не заменяет предметные протоколы (диетические, фармацевтические, поведенческие), а **надстраивается** над ними как мета-уровень, принимающий решение «что и когда предложить конкретному пользователю».

### Контекст

Задача выбора протокола лечения ожирения — **многокритериальная**: эффективность, безопасность, приверженность, стоимость, доступность. Разные клинические рекомендации (EASO 2025, AACE 2025, ADA 2025, NICE 2025, Canadian 2025 — см. COMBINATORY) предлагают детерминированные алгоритмы «если X → то Y». Однако для AI-нутрициолога требуется **вероятностный подход**, учитывающий неполноту данных и индивидуальные предпочтения.

## Алгоритм

### Шаг 0: Инициализация

Загружаются все заполненные поля из цифрового двойника пользователя:

```typescript
interface DigitalTwinState {
  demographics: Partial<Demographics>
  anthropometrics: Partial<Anthropometrics>
  vitals: Partial<Vitals>
  labs: Partial<Labs>
  lifestyle: Partial<Lifestyle>
  genetics: Partial<Genetics>
  medical_history: Partial<MedicalHistory>
  scores: Partial<DigitalTwinScores>
}
```

Загружается каталог протоколов с их индикационными картами (из `_relevance_table.md` и самих протоколов):

```typescript
interface ProtocolIndication {
  protocolId: string                          // TRE_168, MED_DIET, GLP1_RA, ...
  requiredParams: Array<{
    path: string                              // "labs.hba1c_percent"
    threshold: { op: ">=" | "<=" | ">" | "<" | "==" | "between", value: number | [number, number] }
    importance: 1 | 2 | 3                    // 3 = критично, 2 = важно, 1 = уточняет
  }>
  effectSize: number                          // стандартизованный (0..1)
  adherence: number                           // из литературы (0..1)
  evidenceLevel: number                       // high=1.0, moderate=0.7, low=0.4
}
```

### Шаг 1: Расчёт Success Probability для каждого протокола

```
P_success(p) = w₁ × M(p) + w₂ × E(p) + w₃ × A(p) + w₄ × L(p)
```

| Компонент | Обозначение | Вес (по умолч.) | Описание |
|-----------|-------------|-----------------|----------|
| Match quality | M(p) | 0.35 | Насколько заполненные DT-параметры соответствуют пороговым значениям протокола |
| Effect size | E(p) | 0.25 | Стандартизованная величина эффекта (0–1) |
| Adherence | A(p) | 0.20 | Ожидаемая приверженность из литературы |
| Evidence level | L(p) | 0.20 | Уровень доказательности протокола |

#### M(p) — Match Quality

```
M(p) = Σᵢ (wᵢ × matchᵢ) / Σᵢ wᵢ
```

где:
- `matchᵢ` = 1.0 — парам. заполнен и проходит порог
- `matchᵢ` = 0.5 — парам. заполнен, но НЕ проходит порог (протокол всё ещё применим, но менее показан)
- `matchᵢ` = 0.0 — парам. не заполнен (неопределённость)
- `wᵢ` — важность параметра (1–3)

Если ни один из обязательных параметров не заполнен → M(p) = 0.0 (протокол не может быть оценён).

#### E(p) — Normalized Effect Size

```
E(p) = (effect_raw(p) - min_effect) / (max_effect - min_effect)
```

где `effect_raw(p)` — абсолютное значение эффекта (кг потери веса, ΔHbA1c, и т.д.) из протокола, нормированное по всем протоколам в каталоге.

#### A(p) — Adherence Score

Напрямую из протокола: 0.0–1.0 (например, Mediterranean diet: 0.65, GLP-1 RA: 0.55).

#### L(p) — Evidence Level Score

| Уровень | L(p) |
|---------|------|
| high | 1.0 |
| moderate | 0.7 |
| low | 0.4 |
| insufficient | 0.1 |

### Шаг 2: Персонализация (детализация)

```
Personalization(p) = (заполнено_и_проходит_порог(p)) / (всего_параметров_показаний(p))
```

Отображается как дробь `6/8` и как процент `75%`.

### Шаг 3: Ранжирование и показ топ-3

Протоколы сортируются по `P_success(p)` (убывание). Пользователю показываются первые 3 с:

- Название (рус.)
- `P_success` в процентах
- `Personalization` дробью
- Ключевое обоснование (почему именно этот протокол подходит)
- Кнопка «Выбрать» / «Подробнее»

### Шаг 4: Итеративный запрос данных

После показа топ-3 система вычисляет, какие **незаполненные параметры DT** наиболее ценны для уточнения `P_success`.

Для каждого незаполненного параметра `q`:

```
ΔU(q) = |P_success_current - P_success_if_filled(q)|
```

где `P_success_if_filled(q)` — пересчёт вероятностей при гипотетическом заполнении параметра (с оптимистичным допущением, что значение попадёт в порог хотя бы одного протокола из топ-3).

```
Effort(q) = w_time × time_cost(q) + w_money × money_cost(q) + w_access × access_cost(q)
```

| Параметр | time_cost | money_cost | access_cost | Effort (норм.) |
|----------|-----------|------------|-------------|-----------------|
| `anthropometrics.weight_kg` | 0.5 мин | 0 руб | 0 | 0.02 |
| `labs.hba1c_percent` | 15 мин (с визитом) | 500 руб | 0.5 | 0.55 |
| `labs.vitamin_d` | 15 мин | 1500 руб | 0.5 | 0.70 |
| `lifestyle.sleep_hours` | 2 мин | 0 руб | 0 | 0.05 |
| `genetics.apoe` | 30 мин + заказ теста | 5000 руб | 1.0 | 1.00 |
| `vitals.resting_hr_bpm` | 1 мин (wearable) | 0 руб | 0 | 0.03 |

```
Ratio(q) = ΔU(q) / Effort(q)
```

Параметры сортируются по `Ratio(q)` (убывание). Система запрашивает **один** параметр с наивысшим Ratio.

#### UI-паттерн итеративного запроса

```
┌─────────────────────────────────────────────┐
│  Повысим точность подбора                   │
│                                             │
│  📊 HbA1c — наибольшее влияние на подбор    │
│  Если вы укажете HbA1c, точность возрастёт  │
│  на 15% для топ-3 протоколов                │
│                                             │
│  [Ввести вручную] [Из лаборатории] [Пропустить] │
└─────────────────────────────────────────────┘
```

После заполнения → пересчёт `P_success` → обновление топ-3 → запрос следующего параметра.

Критерий остановки: `max(ΔU(q)) < 5%` ИЛИ `Effort(q) > max_effort` (порог, определяемый пользователем или профилем).

### Шаг 5: Выбор пользователя и коммит

Пользователь выбирает один из топ-3 протоколов → система активирует его:

- Записывает выбранный `protocolId` в профиль пользователя
- Устанавливает дату начала и ожидаемую длительность
- Подключает поведенческие интервенции (BEH_01–BEH_07) из `_relevance_table.md`
- Создаёт план мониторинга (какие DT-параметры отслеживать)

## Формулы (сводка)

```
P_success(p) = 0.35·M(p) + 0.25·E(p) + 0.20·A(p) + 0.20·L(p)

M(p) = Σ(wᵢ · matchᵢ) / Σ(wᵢ)    для всех параметров показаний протокола p
  matchᵢ ∈ {1.0, 0.5, 0.0}

E(p) = (effect_raw(p) - min_effect) / (max_effect - min_effect)
  по всем протоколам каталога

Personalization(p) = count(filled_and_meets(p)) / count(total_indications(p))

ΔU(q) = |P_success_top1_current - P_success_top1_if_filled(q)|
  для незаполненного параметра q
  P_success_if_filled(q) — оптимистичный пересчёт (q попадает в порог)

Effort(q) = 0.3·norm_time(q) + 0.4·norm_money(q) + 0.3·norm_access(q)
  norm_* ∈ [0, 1]

Ratio(q) = ΔU(q) / Effort(q)
```

### Настройка весов

Веса могут быть адаптированы под пользователя:

| Профиль | w₁ (Match) | w₂ (Effect) | w₃ (Adherence) | w₄ (Evidence) |
|---------|-----------|-------------|----------------|--------------|
| Default | 0.35 | 0.25 | 0.20 | 0.20 |
| Max результат | 0.20 | 0.50 | 0.10 | 0.20 |
| Max безопасность | 0.30 | 0.10 | 0.30 | 0.30 |
| Новичок | 0.25 | 0.15 | 0.40 | 0.20 |

## Пример работы

### Исходные данные DT

```json
{
  "anthropometrics": { "bmi": 31, "waist_cm": 102 },
  "labs": { "hba1c_percent": 6.2, "ldl_mg_dl": 130, "hdl_mg_dl": 38 },
  "vitals": { "systolic_bp_mmhg": 135 },
  "lifestyle": { "diet": "standard", "sleep_hours": 6, "stress_level_0_10": 7 },
  "medical_history": { "hypertension": "yes" }
}
```

### Расчёт P_success для топ-5 протоколов

| Протокол | M(p) | E(p) | A(p) | L(p) | P_success |
|----------|------|------|------|------|-----------|
| MED_DIET | 0.86 | 0.35 | 0.65 | 1.0 | **0.74** |
| DASH | 0.92 | 0.30 | 0.60 | 1.0 | **0.73** |
| TRE_168 | 0.72 | 0.40 | 0.60 | 1.0 | **0.68** |
| GLP1_RA | 0.50 | 0.85 | 0.55 | 1.0 | **0.65** |
| CBT_WL | 0.40 | 0.25 | 0.55 | 1.0 | **0.52** |

**Топ-3:** MED_DIET (74%), DASH (73%), TRE_168 (68%).

### Итеративный запрос

| Параметр | ΔU | Effort | Ratio |
|----------|-----|--------|-------|
| `labs.crp_mg_l` | 8% | 0.55 | **0.145** |
| `labs.glucose_mg_dl` | 6% | 0.40 | **0.150** |
| `labs.vitamin_d` | 4% | 0.70 | 0.057 |
| `genetics.apoe` | 3% | 1.00 | 0.030 |

Система запрашивает: **`labs.glucose_mg_dl`** (макс. Ratio = 0.150).

После заполнения → пересчёт. Процесс повторяется, пока `max(ΔU) < 5%`.

## Поведенческие интервенции

При активации любого протокола подключаются коды из `_relevance_table.md`:

- **BEH_01** — постановка цели (на 2 и 12 недель)
- **BEH_02** — мотивационное интервью (при сомнениях пользователя)
- **BEH_03** — дневник самоконтроля (ежедневный трекинг)
- **BEH_05** — стимульный контроль (закупка только по списку, убрать триггеры)
- **BEH_07** — релаксация (при высоком стрессе, стресc > 7/10)

## Связь с Цифровым Двойником и интервенциями

| Направление | Ссылка |
|-------------|--------|
| Параметры ЦД | Все 50+ параметров из 9 разделов (см. `_relevance_table.md`). Алгоритм использует любые заполненные поля для расчёта M(p) |
| Интервенции | FD_*, PH_*, SL_*, MN_*, SP_*, M_*, DG_* — назначаются после выбора протокола из его карты в `_relevance_table.md` |
| Поведенческая психология | BEH_01–BEH_07 — универсальные коды, подключаются при активации любого протокола с настройкой под конкретный контекст |
